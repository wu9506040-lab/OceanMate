## 1. 方案整体概述

**一句话**：OceanMate AI 是 OP 商户成功团队的 AI 运营助手体系，由"商户成功 AI 中枢"统领 4 类业务数字员工，覆盖商户选型 / 接入 / 诊断 / 工单 / 知识沉淀 / 协同 6 个环节的全生命周期管理。

### 1.1 业务背景（基于真实搜索数据）

| 项 | 数据 | 来源 |
|----|------|------|
| OP 服务覆盖 | 500+ 支付方式 / 200+ 国家地区 / 6+ 行业 | oceanpayment.com |
| 中国跨境数字支付 2024 | 7.5 万亿元 | 2025 行业报告 |
| 中国跨境数字支付 2025 预测 | 突破 9.4 万亿元（+25% YoY）| 同上 |
| 中国跨境电商进出口 2024 | 2.63 万亿元（+10.8%）| 腾讯新闻 |
| B2B 占比（2025 预测结构）| 65% | 博研咨询 |

### 1.2 为什么不是"AI 客服"而是"运营助手体系"

| 维度 | 跨境商家真实痛点（公开调研）| 传统 AI 客服 | OceanMate |
|------|----------------------|------------|----------|
| 拒付管理 | 59% 商家最关心 / 81% 持卡人拒付仅因没时间申请 | 只能回答"什么是拒付" | 给证据链 + 自动申诉建议 |
| 选型决策 | OP 500+ 支付方式错选代价高 | 答非所问 | 画像匹配推荐 + RAG 证据 |
| 工单协同 | OP 内部"拉群+截图"模式 | 无 | 飞书多维表格自动派单 |
| 知识沉淀 | OP 经验散落各团队 | 无 | 自进化闭环（案例→FAQ→知识库）|

## 2. 整体架构与核心功能模块

### 2.1 三层架构总览

```
┌──────────────────────────────────────────────────────────┐
│  入口层 · 飞书 AI 全家桶                                    │
│    智能伙伴（对话入口）+ 多维表格（工单池/知识库）              │
│    + 妙记（会议沉淀）+ 审批流（SLA 路由）                     │
└──────────────────────────────────────────────────────────┘
                            │
                            ▼
┌──────────────────────────────────────────────────────────┐
│  数字员工层 · 4 类业务 Agent + 1 个中枢                      │
│    • 商户成功 AI 中枢（Orchestrator）—— 意图分流 / 上下文传递 │
│    • Merchant Success Agent (MSA) —— ① 推荐 + ④ 协作采集     │
│    • Payment Diagnosis Agent (PDA) —— ② 诊断（**核心亮点**） │
│    • Ticket Routing Agent (TRA) —— ③ 工单路由              │
│    • Knowledge Evolution Agent (KEA) —— ⑤ 知识自进化        │
└──────────────────────────────────────────────────────────┘
                            │
                            ▼
┌──────────────────────────────────────────────────────────┐
│  基础能力层                                                   │
│    FastAPI + Qwen (DashScope) + RAG + 飞书 API              │
│    Provider 抽象：LLMProvider / VectorStore /              │
│    FeishuConnector / PaymentErrorSource                    │
└──────────────────────────────────────────────────────────┘
```

### 2.2 OP 5 方向 ↔ 4 Agent ↔ 全生命周期映射

| Agent | OP 方向 | 关键能力 | 上游 / 下游 |
|-------|--------|---------|------------|
| **MSA**（数字商户顾问） | ① 推荐 + ④ 协作采集 | 画像匹配 + RAG + 主动反问 | 商户提问 → 转 PDA/TRA |
| **PDA**（支付诊断专家） | ② 诊断（核心亮点） | 多源融合 + 证据链 | MSA / TRA → TRA |
| **TRA**（工单路由专家） | ③ 路由 | 飞书多维表格规则热更新 | MSA/PDA → 飞书工单池 |
| **KEA**（企业知识进化助手） | ⑤ 沉淀 | 案例→FAQ→知识库→RAG 自进化 | TRA 结案 → 知识库 → 下次检索 |

### 2.3 证据链机制（核心亮点 · PDA）

每条诊断结论必带三类证据，可点击逐级追溯：

| 证据类型 | 来源 | 作用 |
|---------|------|------|
| `risk_rule` | 风控规则库 | 解释"为什么被拒" |
| `channel_status` | 通道状态库 | 解释"通道当下状态" |
| `config_snapshot` | 商户配置快照 | 解释"是否配置问题" |

> 设计意图：商户自查 / 工程师复用 / 监管审计均可追溯。

## 3. 核心创新点（5 条）

### 3.1 数字员工定位（模式创新）

- **传统**：AI = 高级客服（问答模式）
- **本方案**：AI = 商户成功团队"数字员工"参与业务决策
- **差异化**：商户成功价值升级——降低风险 / 提升成功率 / 增强长期留存，而非"响应快"

### 3.2 证据链归因（技术 + 流程创新）

- **传统**：AI 给答案无据可查，商家质疑时无据可查
- **本方案**：每条诊断结论附 risk_rule / channel_status / config_snapshot 三类证据
- **差异化**：商家点击自查 / 工程师直接复用排查生产问题 / 监管合规审计可追溯

### 3.3 飞书生态低代码闭环（架构创新）

- **传统**：自建工单系统 + 知识库
- **本方案**：借力飞书 AI 全家桶，多维表格 100 万行/表 + AI 智能标签 + 审批流分发
- **差异化**：运营人员无需开发热更新规则；已验证 "Coze+飞书 3 天落地智能工单"成功路径

### 3.4 知识自进化闭环（模式创新）

- **传统**：FAQ 写完就过时，工程师重复回答相同问题
- **本方案**：案例→FAQ→知识库→下次同类问题自动调用
- **差异化**：自助解决率随时间线性提升（需 OP 真实口径测算）

### 3.5 AtoA Provider 抽象 + MCP 扩展预留（架构创新）

- **现状判断**：2 天冲刺不可能交付完整 AtoA 框架
- **本方案**：把"Agent → Agent"和"Agent → Tool"全部抽象为 Provider 接口，预留 MCP 扩展点
- **差异化**：PoC 不引入完整 MCP 依赖，对接 OP 真实 API 时仅替换 Provider 实现即可

## 4. 落地预期价值（对标真实数据）

> **说明**：所有价值对标基于公开行业数据；**Demo 不做具体百分比承诺**，量化值需在 OP 真实接入后口径测算。

### 4.1 业务价值（对 OP 商户成功团队）

| 价值点 | 对标真实痛点 | 本方案覆盖 | 量化口径（待 OP 真实接入后测算）|
|-------|------------|----------|----------------------------|
| 拒付申诉辅助 | 商家最痛 59% | PDA 证据链 + 申诉路径 | 自助申诉率 ↑ |
| 友好欺诈识别 | 58% | MSA 选型拦截 + PDA 诊断 | 新增欺诈识别覆盖率 ↑ |
| 拒付率降低 | 50% | Merchant Success 选型阶段 | 拒付率 ↓ |
| 工单分流 | OP 内部"拉群+截图"模式 | TRA 飞书多维表格自动派单 | 工单 SLA 达标率 ↑ |
| 知识沉淀 | OP 经验散落各团队 | KEA 自进化闭环 | 自助解决率随时间 ↑ |

### 4.2 商业价值锚点（基于真实市场数据）

- 中国跨境数字支付 2025 预测 9.4 万亿元，+25% YoY
- B2B 占 65%，对应大量需"AI 顾问"的商户决策场景
- OP 自身 500+ 支付方式 / 200+ 国家 / 6+ 行业覆盖度，若商户成功率提升 1pp，可触达数十亿规模支付笔数（具体数字以 OP 内部口径为准）

### 4.3 飞书生态价值（直接对齐赛事命题）

- 工单池 / 知识库 / 审批流全部用飞书原生产品，**非自建**
- 运营人员无需开发即可调整规则（多维表格热更新）
- 与"飞书生态低代码闭环实践"亮点直接对应

## 5. 落地可行性与可推广性

### 5.1 技术可行性

| 维度 | 现状 | 风险 | 缓解 |
|------|------|------|------|
| LLM | Qwen (DashScope) 稳定 | 限流 | 备选 DeepSeek |
| RAG | 飞书多维表格 + 本地向量库 | 性能/规模 | PoC 阶段本地向量库 ok |
| 飞书全家桶 | 多维表格 / 智能伙伴 / 审批流 都有 API | 飞书企业账号未到位 | mock + 文档截图兜底 |
| 数据源 | Demo 占位数据 | OP 真实接口未授权 | Provider 抽象 + 替换占位符即可接入 |

### 5.2 可推广性

| 推广场景 | 可行性 | 说明 |
|---------|--------|------|
| OP 真实环境 | 🟢 高 | Provider 接口已抽象，替换占位符为真实风控/通道/配置 ID 即可 |
| 类似跨境支付公司（PingPong / Airwallex / 连连）| 🟡 中 | 业务模型同构，需定制规则 + SLA |
| 飞书生态内其他行业（电商 / SaaS / 教育）| 🟡 中 | 工单路由 + 知识沉淀模块通用，商户顾问模块需定制 |

### 5.3 与赛事要求对齐（自检）

| 命题要求 | 本方案对齐度 |
|---------|------------|
| 飞书 AI 全家桶深度使用 | ✅ 智能伙伴 / 多维表格 / 妙记 / 审批流 全部使用 |
| A2A / MCP 关键词 | ✅ Provider 抽象 + MCP 扩展点预留 |
| AI 驱动（非"AI 加持"）| ✅ 4 类业务数字员工 + 中枢协同 |
| 跨境 + 商户成功 | ✅ 精准锚定 OP 5 方向 |
| 数据真实可追溯 | ✅ 证据链 + 占位符系统 + 所有数字均带来源 |
