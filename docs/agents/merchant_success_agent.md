# Merchant Success Agent（数字商户顾问）

> 对应 OP 命题方向 ① 支付方式推荐 + ④ 数据协作采集
> 商户入驻期 + 日常运营期的"AI 前台"

> ⚠️ **数据真实性声明（防捏造 · 必读）**
>
> 本文件涉及的具体数字 / 商户画像 / 推荐组合 / 错误码 / 商户 ID 格式等均为 **Demo 占位**：
> - **商户 ID / 订单号格式**（如 `<DEMO_MERCHANT_ID>`）：由 OP 实际系统决定
> - **推荐组合及理由**：Demo 用业内常识表述（基于公开行业报告共识），**上线前需用 OP 内部数据替换**
> - **任何具体百分比 / 增长率 / 渗透率数字，对接真实环境前必须替换为 OP 内部口径**

---

## 1. 职责定位

| 维度 | 内容 |
|------|------|
| 一句话 | **帮商户判断"该开哪些支付方式"，并在商户提问时主动采集上下文** |
| 业务阶段 | 商户入驻 → 日常运营 |
| 上游调用 | 商户成功 AI 中枢（Orchestrator） |
| 下游依赖 | 飞书多维表格（商户画像）、RAG 知识库（支付方式规则）、Qwen LLM |

> **为什么不叫 "Insight Agent"**：避免被理解为"市场分析"，更贴合 OP"商户成功"语境。

---

## 2. 对应 OP 命题方向

| 方向 | 能力 | 输入 | 输出 |
|------|------|------|------|
| ① 支付方式推荐 | 基于画像匹配规则 | 国家、行业、客单价、目标用户 | 推荐支付组合（含推荐理由） |
| ④ 数据协作采集 | 主动反问 + 结构化采集 | 商户原始提问 | 结构化问题档案（商户 ID、国家、渠道、错误码、截图） |

---

## 3. 能力 1：支付方式推荐

### 3.1 输入 Schema

```json
{
  "merchant_id": "<DEMO_MERCHANT_ID>",
  "country_target": "US",
  "industry": "服装电商",
  "avg_order_value": 50,
  "target_user": "年轻消费者"
}
```

### 3.2 输出 Schema

```json
{
  "recommended_methods": ["<基于规则库检索>", "<基于规则库检索>", "<基于规则库检索>"],
  "rationale": [
    "<引用规则库条目 1>",
    "<引用规则库条目 2>",
    "<引用规则库条目 3>"
  ],
  "expected_conversion_uplift": "需以 OP 内部数据口径测算（Demo 阶段不输出具体百分比）",
  "evaluation_metrics": ["自助开通率", "上线后首月支付成功率"]
}
```

> **关键**：推荐组合的"具体支付方式"和"推荐理由"在 Demo 阶段以占位符呈现；上线前需用 OP 规则库 + OP 内部数据替换。所有推荐必须可追溯到规则库条目（RAG 证据）。

### 3.3 技术实现

- **RAG 检索**：从支付方式规则库（`docs/data/payment_methods_rules.json`）检索匹配项
- **LLM 推理**：Qwen 模型综合画像 + 规则输出推荐 + 推荐理由
- **规则兜底**：若 LLM 输出与规则冲突，以规则为准（防幻觉）

## 4. 能力 2：数据协作采集

### 4.1 场景

商户在飞书智能伙伴提问：
> "我美国站今天好几单支付失败了，怎么办？"

### 4.2 AI 主动采集逻辑

不是立刻回答，而是先反问：

> AI："为了帮你快速定位，我需要你提供以下信息：
> 1. 商户 ID（具体格式由 OP 实际系统决定）
> 2. 受影响的订单号
> 3. 错误码（如果有）
> 4. 失败时间范围
> 5. 截图（可选）"

> **设计意图**：让商户从"问答模式"切换到"协作模式"，与 OP 命题"数据协作采集"挑战直接对应——这**不是聊天而是结构化采集**。

### 4.3 输出 Schema

```json
{
  "problem_record": {
    "merchant_id": "<DEMO_MERCHANT_ID>",
    "country": "US",
    "channel": "<DEMO_CHANNEL>",
    "error_codes": ["<DEMO_ERROR_CODE>"],
    "affected_orders": ["<DEMO_ORDER_ID>"],
    "evidence": ["<screenshot_url>"]
  },
  "next_agent": "Payment Diagnosis Agent"
}
```

## 5. 与其他 Agent 协作

```
商户提问 ──► Merchant Success Agent ──► 结构化问题档案
                                              │
              ┌───────────────────────────────┤
              ▼                               ▼
    Payment Diagnosis Agent         Ticket Routing Agent
        （诊断归因）                  （工单分派）
```

> 注意：方向 ④（数据协作采集）由本 Agent 与中枢共同承担——Agent 负责**结构化采集**，中枢负责**意图分流**。

---

## 7. 边界

- ❌ 不承诺精确转化率数字
- ❌ 不接入真实支付通道（PoC 阶段用规则库）
- ❌ 不编造任何"具体真实"的渗透率 / 增长率 / 行业数据
- ✅ 所有推荐必须可追溯到规则库条目（RAG 证据）
- ✅ 数据协作采集必须结构化产出 `problem_record`
- ✅ 所有"具体百分比 / 增长率"必须可追溯到规则库 / OP 内部数据
