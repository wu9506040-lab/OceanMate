# Knowledge Evolution Agent（企业知识进化助手）

> 对应 OP 命题方向 ⑤ 知识沉淀
> 商户生命周期最后一个闭环：让经验沉淀为可复用资产

> ⚠️ **数据真实性声明（防捏造 · 必读）**
>
> 本文件涉及的具体错误码 / 规则号 / 商户 ID 等均为 **Demo 占位符**：
> - 案例 `error_code` / `root_causes` 在演示中以 `<DEMO_ERROR_CODE>` / 占位规则说明呈现
> - case_id 由系统按规则生成（Demo 用 `C<YYYYMMDD>xxx` 格式），**非真实编号**
> - **任何缺失 `_demo` 后缀或 `<xxx_demo_xxx>` 占位符的具体数字，对接真实环境前必须替换**

---

## 1. 职责定位

| 维度 | 内容 |
|------|------|
| 一句话 | **单次问题解决后自动总结为结构化案例 → 生成 FAQ → 入企业知识库 → 下次同类问题中枢自动调用，业务能力可自我进化** |
| 业务阶段 | 全生命周期（异常处理后必有） |
| 上游调用 | Ticket Routing Agent（工单结案后）/ Payment Diagnosis Agent（直接路径） |
| 下游依赖 | 企业知识库（飞书多维表格）、RAG 索引 |

> **设计的核心意图**：让 OP 商户成功团队**踩过的坑**变成 AI 的**肌肉记忆**——下次同类问题自动解决，**自助解决率随时间线性提升**。

---

## 2. 对应 OP 命题方向

| 方向 | 能力 | 价值 |
|------|------|------|
| ⑤ 知识沉淀 | 案例→FAQ→知识库→下一次自动调用 | 让知识库与商户问题同步进化 |

---

## 3. 输入

```json
{
  "ticket": { ... },        // Ticket Routing 生成的工单（结案状态）
  "diagnosis": { ... },     // Payment Diagnosis 输出
  "resolution": {           // 人 / AI 协作的处理结果（结案时填）
    "actions_taken": ["<Demo 操作 1>", "<Demo 操作 2>"],
    "outcome": "支付成功率恢复到正常水平（具体数值需以 OP 内部口径为准）",
    "duration_hours": "<DEMO_DURATION>"
  }
}
```

---

## 4. 处理流程

```
结案工单 ──► 案例特征抽取 ──► 结构化案例 ──► FAQ 草稿
                                              │
                                              ▼
                                       知识库写入（飞书多维表格）
                                              │
                                              ▼
                                    RAG 索引更新
                                              │
                                              ▼
                                    下次同类问题自动调用
```

### 4.1 案例特征抽取

从 resolution 中抽取：
- **问题类型**（payment_failed / chargeback / ...）
- **国家 / 渠道 / 错误码**
- **根因组合**（具体风控规则 + 3DS 配置等多重因素，详见 Payment Diagnosis 输出）
- **解决路径**（开启 3DS + 风控白名单等组合，Demo 占位）
- **耗时 / 结果**

### 4.2 结构化案例 Schema

```json
{
  "case_id": "C<YYYYMMDD>xxx",
  "problem_type": "payment_failed",
  "country": "BR",
  "channel": "Visa",
  "error_code": "<DEMO_ERROR_CODE>",
  "root_causes": [
    "<Demo 风控规则命中（具体 ID 见 Payment Diagnosis evidence）>",
    "<Demo 3DS 配置相关>"
  ],
  "resolution_path": [
    "<Demo 操作 1>",
    "<Demo 操作 2>"
  ],
  "duration_hours": "<DEMO_DURATION>",
  "outcome": "success",
  "faq_summary": "<具体国家 + 渠道 + 错误码下，常见根因 + 处理路径的通用化总结>"
}
```

> **所有 `<DEMO_xxx>` 占位符在真实场景下替换为 OP 实际错误码 / 规则 ID / 配置项**。

### 4.3 FAQ 草稿（自动生成，人工 review 上线）

```
Q: <country> 渠道订单支付失败，错误码 <DEMO_ERROR_CODE>，是什么原因？
A: 通常是多重因素叠加（具体取决于错误码、风控规则、配置等）：
   1. <country> 渠道触发风控规则（具体规则 ID 详见 Diagnosis 输出）
   2. 3DS 配置相关问题（按当地合规要求可能强制，具体由 OP 规则决定）

   建议处理：
   1. 检查对应区域的 3DS 配置（Merchant Console → 风控设置）
   2. 提单风控团队复核规则是否需要白名单（附商户 ID + 订单号）
   3. 参考知识库同类型历史案例
```

> Demo FAQ 模板：具体国家 / 渠道 / 错误码 / 规则均为占位符，真实环境接入时由 Knowledge Evolution Agent 根据实际结案数据自动生成。

## 5. 知识库回灌与下一次调用

| 阶段 | 内容 |
|------|------|
| 入库 | 写入飞书多维表格「OP 企业知识库」表 |
| 索引 | 自动更新向量索引（RAG 检索；PoC 阶段可用本地向量库代替） |
| 下次调用 | Merchant Success / Payment Diagnosis 在 RAG 检索时自动命中此案例 |
| 命中效果 | 自助解决率随时间提升，新商户问题接近 0 等待 |

## 7. 闭环：商户问题 → AI 解决 → 知识增长 → 下次更快解决

```
                  商户问题
                     │
                     ▼
         ┌───────────────────────┐
         │ AI 解决（4 Agent 协同）│
         └───────────────────────┘
                     │
                     ▼
              知识沉淀（Knowledge Evolution）
                     │
                     ▼
              下次更快解决（自进化）
                     │
                     ▼
        商户风险↓、成功率↑、留存↑
```

> 商户成功团队的核心价值**不是解决问题**，而是：
> - **降低商户运营风险**
> - **提升支付成功率**
> - **增强商户长期留存**

## 9. 边界

- ❌ 不替换人工 FAQ 审核（自动生成，人工 review 上线）
- ❌ 不强制全量覆盖历史案例（增量更新）
- ❌ **不编造任何"具体真实"的规则号 / 错误码**
- ✅ 案例写入知识库全流程可追溯
- ✅ 知识库条目可被任一 Agent 调用
- ✅ 所有更新走 RAG 索引机制，便于灰度发布
