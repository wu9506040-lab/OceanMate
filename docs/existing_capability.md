# AI 能力基础（existing_capability.md）

> 本文件锁定项目**底层 AI / 工程能力**的来源说明。
> 用于比赛评审快速了解：为什么我们有信心在 3 周内出 MVP。

---

## 1. 一句话定位

本项目（OceanMate AI）的工程能力**并非从零搭建**，而是从已有的企业级 AI Agent 实践**精准迁移到跨境支付商户成功场景**。

**AI 能力来源：已有企业级智能客服 Agent 实践（已落地）。**

> **重要表述边界**：本文写"AI 能力迁移"，**绝不写"复用客服代码"**——前者是商业叙事，后者是技术细节，二者混用会让方案显得像"项目迁移"而非"产品创新"。

---

## 2. 已有 AI 能力清单

| # | 能力 | 已落地场景 | 本项目复用方式 |
|---|------|----------|--------------|
| 1 | **LLM 编排（Qwen / DashScope 兼容）** | 智能对话 | OceanMate 所有 Agent 的推理引擎 |

| 5 | **FastAPI 工程化后端** | 客服对话服务 | OceanMate 后端框架 |
| 6 | **会话管理与上下文跟踪** | 多轮对话 | 商户提问时的上下文采集 |

| 8 | **Prompt 模板与配置化** | 客服话术管理 | Prompt 集中管理（按 Agent 拆分） |

## 3. 能力迁移路径

```
已有能力（企业级智能客服 Agent 实践）
            │
            ├─ LLM 编排 ───────────►  OceanMate 所有 Agent 的推理基础
            ├─ RAG 检索 ───────────►  支付方式规则 / 错误码库 / 知识库
            ├─ 向量数据库 ─────────►  企业知识库索引
            ├─ 多 Agent 编排 ──────►  商户成功 AI 中枢
            ├─ FastAPI 工程化 ─────►  4 Agent 接口
            ├─ 飞书 webhook ───────►  智能伙伴 / 多维表格集成
            └─ Prompt 配置化 ──────►  4 Agent 的角色 Prompt 集中管理
                │
                ▼
        OceanMate AI 跨境商户成功助手
```

> 迁移路径中每一条都是已有能力的直接复用，不是"借鉴"也不是"参考"——是**原能力的场景化精准定位**。

## 5. 不引入的依赖（CLAUDE.md §2 禁止）

| ❌ 不引入 | 原因 |
|-----------|------|
| Kafka / Milvus / Elasticsearch | 单体架构 PoC 不需要 |
| **新的数据库** | 飞书多维表格替代结构化数据；妙记替代非结构化数据 |
| 复杂中间件 | 保持单体架构 + 飞书生态 |
| 新的 LLM SDK | 复用已有 Qwen / DashScope 兼容能力 |
| 自建向量库 | Qdrant 已落地，复用即可 |

## 7. 与原项目关系（重要 · 评审必看对照表）

| 维度 | 已有 AI Agent 实践 | OceanMate AI（本项目） |
|------|-----------------|---------------------|
| 业务定位 | 通用客服 | 跨境支付商户成功 |
| Agent 体系 | 智能对话 Agent | 商户成功 AI 中枢 + 4 业务 Agent |
| 数据底座 | MySQL + Qdrant | 飞书多维表格（不引入新 DB） |
| 协同模式 | 单一对话流 | 多 Agent 协作 + 飞书生态 |
| 落地形式 | 智能客服系统 | 飞书智能伙伴 + 多维表格 + 妙记 |

> **重要声明**：OceanMate AI 是已有 AI Agent 能力**在 OP 商户成功场景下的精准迁移**，而非从零搭建。这是**3 周能出 MVP 的关键原因**——70% 能力已有，新增 30% 按 WBS 节奏推进。

## 8. AtoA / MCP 在本项目中的降级表述

为避免给评审"工程过度"印象，本项目对 AtoA / MCP 采用**降级表达**：

| ❌ 不这样写 | ✅ 这样写 |
|------------|----------|
| "已实现 AtoA 协议" | "采用 Agent 任务协作机制" |
| "MCP 对 MCP 调用" | "系统采用 Agent 任务编排机制，通过**标准化工具接口**支持未来 MCP 扩展" |
| "AtoA 是核心卖点" | "AtoA 是工程概念，非商业卖点；商业卖点是 OP 业务闭环" |

> 关键词保留（让评审知道你知道这些），但**实现压力没有**（不假装 2 天能交付完整的 AtoA 框架）。

## 9. 评审视角自检

| 评审问题 | 答案 |
|---------|------|
| 为什么 3 周能出 MVP？ | 已有 LLM/RAG/FastAPI/多 Agent 编排能力，新增仅是支付规则库 + 飞书集成 |
| 技术栈为什么不引入新依赖？ | 复用已有组件；飞书多维表格替代新数据库 |
| 单人冲刺可行性？ | 已有经验覆盖 70%；新增 30% 按 WBS 节奏推进 |
| 与学生"做个聊天机器人"的本质区别？ | 不是"做一个 AI 工具"，是"把已有企业 AI 能力迁移到 OP 商户成功场景" |
| 未来如何平滑升级？ | 模拟数据 → 真实 API 时，证据链 / 路由规则 / RAG 索引机制**结构不变**，仅替换数据源 |
