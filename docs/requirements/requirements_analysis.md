## 1. 业务背景

### 1.1 行业趋势：从"通道竞争"到"服务能力竞争"

跨境支付行业经历三个阶段：

| 阶段 | 焦点 | 代表能力 |
|------|------|----------|
| 1.0（2010-2018）| 通道接入 | 支持更多卡组织 / 国家币种 |
| 2.0（2019-2024）| 风控与合规 | Stripe Radar 等 AI 风控 / 拒付拦截 [1] |
| **3.0（2025+）** | **商户服务能力** | **选型 / 接入 / 诊断 / 协同 / 知识全链路** |

**关键证据**：
- Airwallex 跨境消费研究显示，**54%** 消费者预计增加跨境购物、**77%** 因缺偏好支付方式放弃 [2]
- 中国跨境数字支付 2024 年 7.5 万亿元 → 2025E 突破 9.4 万亿元（+25% YoY）[3]
- 头部企业 Stripe Radar 等仍聚焦交易风控，**商户接入 / 异常定位 / 跨团队协作等服务流程仍高度依赖人工经验** [4]

### 1.2 OP 命题定位

OP（Oceanpayment）业务覆盖 **500+ 支付方式 / 200+ 国家地区 / 5+ 行业**（跨境外贸 / 旅游航空 / 软件游戏 / 数字版权 / 教育培训）[5]。商户问题常涉及 **支付渠道 + 风控策略 + 配置规则 + 订单数据** 等多因素叠加，**传统人工支持模式难以随业务规模持续扩展**。

### 1.3 核心痛点（来自公开行业反馈）

| 痛点 | 数据 | 业务影响 |
|------|------|----------|
| 退款争议 | 网经社 2024 报告：**跨境电商"退款问题"投诉 TOP 1，占比 20.00%** [6] | 商户申诉成本高 / OP 客服压力大 |
| 友好欺诈 | 同上：「任意仅退款」占 **13.60%** [6] | 商户资金损失 / 风控规则需持续优化 |
| 工单协同 | OP 内部"拉群 + 截图"模式 | 信息传递损耗 / 跨团队响应慢 |
| 知识沉淀 | OP 经验散落各团队 | 新员工上手慢 / 同类问题重复处理 |

## 2. 用户画像（4 类核心用户）

| # | 角色 | 关注点 | 当前痛点 | 与 OceanMate 交互方式 |
|---|------|-------|---------|---------------------|
| ① | **商户主**（决策层） | 业务增长 / 支付成功率 | 选错支付方式 → 流失订单 | 飞书智能伙伴对话 · 商户顾问 Agent |
| ② | **商户运营**（执行层）| 日常运营 / 异常处理 | 故障定位耗时 / 跨团队沟通 | 飞书智能伙伴 + 多维表格 · 支付诊断 + 工单路由 Agent |
| ③ | **OP 客服**（CSM 团队）| 工单处理 / SLA | 重复问题高频 / 知识检索难 | 飞书多维表格 · 工单路由 + 知识优化 Agent |
| ④ | **OP 研发 / 风控**（内部支撑）| 根因定位 / 规则迭代 | 截图排查 / 规则人工更新 | 飞书审批流 · 知识优化 Agent 沉淀 + 反馈 |

> **关键洞察**：OP 服务体系天然分两层（商户侧 + 内部侧），OceanMate 不做单端客服系统，而做 **"对商户是 AI 合伙人，对 OP 是数字员工体系"** 的双向助手。

## 3. 用户故事（8 条 · As a ... I want ... So that ...）

### 商户侧（3 条）

| ID | 用户故事 | 对应 Agent | 优先级 |
|----|---------|-----------|--------|
| US-1 | **作为商户主**，我希望在接入 OP 前能根据我的国家 / 行业 / 目标市场获得支付方案推荐，**以便**降低选型试错成本。 | 商户顾问 Agent | P0 |
| US-2 | **作为商户运营**，我希望在支付失败时能拿到"错误码 + 渠道状态 + 配置 + 案例"四维分析，**以便**自助定位问题或快速提交工单。 | 支付诊断 Agent | **P0（Demo 核心）** |
| US-3 | **作为商户运营**，我希望工单状态对商户可查、处理进度有 SLA 提醒，**以便**减少"催单"沟通成本。 | 工单路由 Agent | P1 |

### OP 内部侧（5 条）

| ID | 用户故事 | 对应 Agent | 优先级 |
|----|---------|-----------|--------|
| US-4 | **作为 OP 客服**，我希望自动收到工单分派通知（含问题类型 / 紧急度 / 证据链），**以便**直接处理而非先做"问题归类"。 | 工单路由 Agent | P0 |
| US-5 | **作为 OP 客服**，我希望处理工单时能检索历史相似案例 + 标准话术，**以便**提升处理一致性。 | 知识优化 Agent | P1 |
| US-6 | **作为 OP 客服**，我希望工单关闭后处理过程自动沉淀为 FAQ / 知识条目，**以便**下次同类问题自动命中。 | 知识优化 Agent | P1 |
| US-7 | **作为 OP 研发 / 风控**，我希望从工单里看到证据链 + 趋势，**以便**驱动规则迭代和接口优化。 | 支付诊断 Agent + 知识优化 Agent | P2 |
| US-8 | **作为 OP 管理者**，我希望看到工单分布 / 处理时长 / 知识命中率的看板，**以便**评估 AI 助手 ROI。 | 知识优化 Agent（数据看板） | P2 |

> **优先级说明**：P0 = PoC 必交付 · P1 = Phase 3 落地 · P2 = 后续迭代

## 4. 功能需求（按 Agent 拆解）

### 4.1 商户顾问 Agent（MSA）

| 能力 | 描述 | 优先级 |
|------|------|--------|
| 画像采集 | 通过对话采集国家 / 行业 / 目标市场 / 客单价 | P1 |
| 方案推荐 | RAG 检索 OP 500+ 支付产品 + 历史案例，输出 Top-3 推荐 | P1 |
| 友好欺诈预判 | 识别"高退款率行业"商户，给出风控配置建议 | P2 |

### 4.2 支付诊断 Agent（PDA）⭐ Demo 核心

| 能力 | 描述 | 优先级 |
|------|------|--------|
| 错误码归因 | 融合错误码 / 渠道状态 / 配置规则 / 历史案例 4 类证据 | **P0（PoC）** |
| 证据链输出 | 每条诊断附 `risk_rule` / `channel_status` / `config_snapshot` 三类可追溯证据 | **P0（PoC）** |
| 根因 + 建议 | 输出问题类型 + 根因列表 + 推荐处理动作 | **P0（PoC）** |
| 下一跳触发 | 诊断完成后自动触发工单路由 Agent | **P0（PoC）** |

### 4.3 工单路由 Agent（TRA）

| 能力 | 描述 | 优先级 |
|------|------|--------|
| 问题分类 | 按诊断 Agent 输出自动归类（业务 / 技术 / 财务 / PSP）| P0 |
| 飞书派单 | 通过飞书多维表格 + 审批流按规则自动派单 | P0 |
| SLA 跟踪 | 工单超时提醒 + 进度同步给商户 | P1 |

### 4.4 知识优化 Agent（KEA）

| 能力 | 描述 | 优先级 |
|------|------|--------|
| 工单沉淀 | 处理完成后自动抽取 FAQ 条目 | P1 |
| 知识库更新 | 写入飞书多维表格（结构化）+ 妙记（非结构化）| P1 |
| 自进化检索 | 下次同类问题自动 RAG 命中 | P1 |

## 5. 非功能需求

| 类别 | 要求 | 说明 |
|------|------|------|
| **数据真实性** | 所有量化数据可追溯至公开来源 | 不编造 / 不夸大 / 不承诺未验证效果 |
| **合规** | 飞书生态原生（不引入新数据库） | 遵循 OP 数据安全规范 |
| **可推广** | 模块化设计可接入 OP 支付 / 订单 / 风控数据 | PoC 阶段 Provider 抽象预留扩展 |
| **工程纪律** | 单 Agent Scope Lock / 验证分级 / AtoA Provider 抽象 | 见 `CLAUDE.md` §3-5 |
| **Demo 友好** | 终端 `python run_demo.py` 即可出诊断结果 | 无需 API Key / 无需启动服务 |

## 6. 验收标准（5 项可验证指标）

| # | 指标 | 验证方式 | 通过判据 |
|---|------|---------|---------|
| 1 | **诊断可追溯** | `run_demo.py` 输出 `evidence_chain` 字段 | 每条根因可定位到 `risk_rule` / `channel_status` / `config_snapshot` |
| 2 | **错误码覆盖** | 切换 `--error-code` 跑 5 个 mock 错误码 | ERR_DEMO_RISK_BLOCK / 3DS_REQUIRED / CHARGEBACK / REFUND_REJECTED / CHANNEL_DOWN 全部命中 |
| 3 | **架构图渲染** | 浏览 `docs/architecture/agent_architecture.md` 与 `business_flow.md` | Mermaid 图无交叉 / 无对角线 / 渲染正常 |
| 4 | **接口签名稳定** | 启动 FastAPI 调 `/api/diagnose` | 返回 200 + DiagnoseResponse JSON |
| 5 | **Provider 可替换** | 替换 `evidence_store.py` 三个查询方法为 OP 真实 API | 接口签名不变 / evidence.id 占位符替换为 OP 真实 ID 即可 |

> **效果指标**：退款申诉处理时长 / 工单分流准确率 / 知识库命中率等业务指标**需 OP 真实接入后口径测算**，本方案不承诺具体百分比。
