# OP 商户成功业务闭环定义（merchant_success.md）

> 本文件锁定 OP 命题中 5 类核心业务方向，作为 OceanMate AI 4 类 Agent 的需求来源。
> 一行方向 = 一个 Agent 的责任 = 一个 Demo 验证点。

---

## 1. 一句话定位

**OceanMate AI = OP 商户成功团队的 AI 数字员工体系。**

OP 商户成功团队（CSM）日常工作覆盖商户全生命周期：选型咨询 → 接入协助 → 异常诊断 → 工单协同 → 经验沉淀。本方案将每一类高频协作转化为 AI 数字员工的能力，使商户成功从"被动响应"升级为"主动运营"，让 CSM 团队从成本中心升级为利润中心。

> **边界**：本方案是"AI 数字员工体系"，不是"AI 客服机器人"；是"重构协作模式"，不是"做一个工具"。

---

## 2. OP 命题五大核心方向 ↔ OceanMate 4 类 Agent 对照表

| # | OP 命题核心方向 | 真实业务场景 | 对应 Agent | 必含能力 |
|---|---------------|-------------|-----------|---------|
| ① | **支付方式推荐** | 商户入驻初期，不知道目标国家应开通哪些支付方式 | **Merchant Success Agent** | 商户画像、规则库、RAG |
| ② | **支付失败 / 拒付 / 退款诊断** | 支付失败、拒付、退款、对账差异归因不清 | **Payment Diagnosis Agent**（Demo 核心） | 多源融合、因果归因、证据链 |
| ③ | **工单智能路由** | 问题需业务 / 技术 / 财务 / PSP 多团队分别响应 | **Ticket Routing Agent** | 问题分类、部门映射、飞书审批 |
| ④ | **数据协作采集** | 首次响应要商户反复提交订单号 / 截图 / 日志 | **Merchant Success Agent + 商户成功 AI 中枢** | 主动反问、结构化问题档案 |
| ⑤ | **知识沉淀** | FAQ 更新滞后，新人重复踩同一坑 | **Knowledge Evolution Agent** | 案例总结、FAQ 生成、知识库回灌 |

> **关键设计**：4 类 Agent 对应 5 个方向——"Merchant Success Agent" 同时承载方向 ① 和 ④，避免 Agent 数量堆砌。

---

## 3. 商户成功 AI 中枢（Orchestrator）

| 维度 | 内容 |
|------|------|
| 角色 | 上层入口 / 任务路由 / 状态编排 |
| 输入 | 商户从飞书智能伙伴发起的自然语言咨询 |
| 输出 | 调度的 Agent 协同结果，反写飞书多维表格 / 妙记 / 群消息 |
| 关键能力 | **主动采集上下文**、意图识别、Agent 选择、结果聚合、状态回写 |
| 与 Agent 区别 | 中枢 = 编排；Agent = 执行 |

> 中枢"主动采集上下文"是 OP 命题"数据协作采集"挑战的直接回应——AI 不替代人，但参与商户信息的首次结构化采集。

---

## 4. 商户生命周期管理（AI 参与的 4 阶段）

```
[商户入驻] ──────► [日常运营] ──────► [异常处理] ──────► [长期增长]
    │                  │                  │                  │
    ▼                  ▼                  ▼                  ▼
Merchant           Merchant           Payment            Knowledge
Success            Success            Diagnosis          Evolution
Agent              Agent              Agent              Agent
    │                  │                  │                  │
  选型咨询         接入协助          异常归因          经验沉淀
```

AI 在每个阶段都是一个具体动作的执行者，**全链路参与商户生命周期**而非被动等待询问。

---

## 5. 关键边界（不许越线）

- ❌ 不承接 OP 内部财务 / 合规 / 法务系统对接（超出 PoC 范围）
- ❌ 不承诺"自动提升商户成功率 XX%"（量化要可验证，非编造）
- ❌ 不堆 Agent 数量——4 类业务 Agent 已是上限
- ✅ 所有数据接入走飞书多维表格 / 妙记，不引入新数据库
- ✅ 所有量化指标写"预期可"+"后续以 XX 指标评估"
- ✅ AtoA / MCP 不作为对外卖点，仅作"标准化工具接口预留 MCP 扩展"的内部工程表述
